## 三、根因一：把 Claude 的报错当成网站的报错

### 现象

对话中反复出现 `Server error`。因为项目本身就是一个 Web 服务，"server error" 这个词天然指向"我的网站崩了"，于是开始查服务端日志、查路由、查数据库——**查了一个不存在的 bug**。

### 真相

那是 **Claude 自己后端的报错**：单次操作载荷过大，Claude 的服务处理不了，返回 Server error。和被开发的网站毫无关系。

### 为什么会误判

1. **字面陷阱**：术语撞车。开发 Web 应用时，"server error" 有一个极强的默认解释。
2. **上下文偏置**：正在写后台代码时报错，最自然的假设就是"刚写的代码有问题"。
3. **AI 助手带头误判**：Claude 自己也把它当成了网站问题去排查，用户跟着一起找错方向。工具的误判会被用户默认信任，因此**代价加倍**。

### 可操作的判据

> **报错时第一个问题不是"哪里错了"，而是"谁在报错"。**

| 判据 | 你的网站在报错 | Claude/工具链在报错 |
|---|---|---|
| 服务端日志有记录吗 | ✅ 有 | ❌ 无 |
| 浏览器控制台/Network 有吗 | ✅ 有 HTTP 状态码 | ❌ 无 |
| 网站本身还能访问吗 | ❌ 通常不能 | ✅ 完全正常 |
| 错误只出现在对话里吗 | ❌ | ✅ |
| 重试同样的操作 | 稳定复现 | 换个小操作就好了 |

**30 秒验证法**：报错时立刻 `curl` 一下站点。返回 200 = 网站活得好好的，错误不在你的代码里。

## 四、根因二：单次写入超大文件

### 核心区分（本复盘最重要的一条）

> **不是"文件不能大"，是"单次写入不能大"。**

反证就在同一个仓库里：

| 文件 | 当前体积 | 是否出事 | 为什么 |
|---|---|---|---|
| `src/routes/admin.js` | **22,515 字符** | ❌ 从未 | 一次次 Edit **累积长出来**的 |
| `src/db.js` | 19,233 字符 | ❌ 从未 | 同上 |
| `views/admin.ejs`（事故期） | 目标 ≈ 44,000 字符 | ✅ 反复崩 | 试图**一次写完** |

`admin.js` 比当年出事的 admin.ejs 还大，却安然无恙。**决定成败的是单次操作的载荷，不是文件的最终体积。**

### 阈值经验

- 单次 Write 控制在 **10k 字符以内**是安全区
- 超过 **15k** 开始有风险
- 接近 **30k+** 基本必崩

### 修复方式

把单体 admin.ejs 拆成 5 片，逐个写入：

```
views/admin.ejs              14,647 字符  ← 骨架 + include
views/partials/admin-agent.ejs   10,601
views/partials/admin-content.ejs  8,223
views/partials/admin-pages.ejs    3,303
views/partials/admin-kb.ejs       2,881
```

### 意外收获：约束逼出了更好的架构

分片 include **不只是绕过工具限制**——它本身就是更好的结构。本次会话改后台时的直接受益：改内容管理只需动 `admin-content.ejs`（8k），完全不必碰其它部分。如果还是 44k 的单体文件，每次小改动都要在巨石里定位，出错概率和成本都高一个量级。

**工具的约束有时是伪装的设计建议。** 一个"必须拆开才能写"的文件，通常本来就该拆开。

## 五、根因三：僵尸进程，以及它的"解药"

### 症状链

```
上次的 node 进程没退干净
  → 端口 8201 被占
    → 新服务起不来 / 起来了但你连的是旧的
      → 改了代码没生效
        → 看起来像"代码坏了"  ← 又一次归因错误
```

僵尸进程的真正危害不是占资源，而是**制造伪症状、污染判断**——它让你怀疑代码，而问题在环境。

### 现场抓到的反面教材

本次会话中反复使用的清理命令：

```bash
taskkill //F //IM node.exe    # ← 危险
```

进程快照显示，这台机器上当时运行的 node 进程是：

```
"D:\Program Files (x86)\Tencent\微信web开发者工具\node.exe"
```

**`taskkill //IM node.exe` 杀的是全机器所有 node 进程**，包括微信开发者工具这类完全无关的第三方应用。它不区分"我的开发服务器"和"别人的程序"。用这种方式清僵尸，等于用手雷灭蚊子。

### 更隐蔽的连带伤害：重启污染了测试

本次会话真实发生的一次：

1. 重启开发服务器（清进程）
2. → 内存态 session 全部失效
3. → 浏览器里的登录态没了
4. → 所有 API 返回 403
5. → 测试结果：**「前台已隐藏 ✓」「已清理 ✓」**

看起来通过了。真相是：**什么都没创建，所以"看不到"当然成立**——这是**空过的假阳性**。如果不是那个"全 ✗ 但某几项 ✓"的诡异组合引起警觉，这个错误结论就被当成"测试通过"了。

### 正确做法

```bash
# ✗ 别这么干
taskkill //F //IM node.exe

# ✓ 按端口精确定位再杀
netstat -ano | grep 8201          # 找到占用者的 PID
taskkill //F //PID <那个PID>       # 只杀它

# ✓ 更好：换个端口，根本不用杀
PORT=8299 node server.js
```

## 六、本次会话新抓到的坑（活体样本）

事故期的坑已成历史，但同类问题仍在持续发生。以下是**本次会话内**真实踩到的：

| 坑 | 症状 | 教训 |
|---|---|---|
| **假阳性测试** | 未登录状态下测试，接口全 403，"没找到"被当成"已删除✓" | 通过必须有**正向证据**（创建了什么、返回了什么 ID），不能靠"找不到"反推 |
| **Git Bash 路径改写** | `/tmp/x` 被改写成 `E:\tmp\x` | Windows 上的 Git Bash 会转换路径，用项目内相对路径或 `$TEMP` |
| **反引号被执行** | `` `GET /api/x` `` 在 `bash -c` 里被当命令替换执行，把 CHANGELOG 写坏 | 含反引号的文本用 heredoc 或 Write 工具，别塞进 shell 字符串 |
| **env 变量位置** | `node -e "..." VAR=x` 无效（须写在**前面**） | bash 的 env 前缀语法是 `VAR=x cmd`，写后面会被当 argv |
| **误建空数据库** | `better-sqlite3('data/alan.db')` 对不存在的路径**直接建新库**，而真实路径是 `app.db` | 打开数据库前先确认路径；只读用 `{readonly:true}` 可避免误建 |

## 七、沉淀：规则（已写入全局 CLAUDE.md）

文档会被遗忘，规则每个会话都加载。以下是本复盘转化出的**可执行规则**：

### 规则 A：先定位报错来源，再排查

出现 `Server error` / 工具报错时，**第一步是判断是谁在报错**，不是查代码：
- 立刻 `curl` 站点，返回 200 → 网站没事，错在工具链
- 服务端日志无对应记录 → 不是你的代码
- 换个更小的操作就成功 → 是载荷问题，不是逻辑问题

### 规则 B：单次写入不超过 10k 字符

- 文件可以大，**单次 Write/Edit 的载荷必须小**
- 预计超过 10k 的文件，**先设计分片结构再动手**（partial / 模块 / include）
- 大文件靠**多次小 Edit 累积**，不靠一次写完
- 反复出现工具报错时，第一反应是"我的操作是不是太大了"

### 规则 C：禁止按进程名批量杀进程

- **禁用** `taskkill //F //IM node.exe`（误伤全机器的 node 应用）
- 用 `netstat -ano | grep <端口>` 定位 PID，精确 kill
- 优先换端口，而不是杀进程

### 规则 D：测试的"通过"必须有正向证据

- 断言必须基于**存在性证据**（返回的 ID、创建的记录、数据库计数）
- 禁止用"找不到 / 看不到"单独作为通过依据——**未登录、没创建成功、请求根本没发出去，都会让"看不到"成立**
- 重启服务后，先验证登录态/前置条件仍然有效，再跑测试

### 规则 E：小步快跑，禁止憋大招

- 每完成一个可验证的小单元就提交，**不要攒到"全部完成"**
- 39 小时零提交 vs 3 小时 8 次提交，是同一个项目的两种命运
- 大任务先拆解，每步都要能独立验证、独立提交
